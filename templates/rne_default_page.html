Guidelines Headline
What to consider:​
  • This is the single most important piece of information – must make sense to a non-subject matter expert. ​
  • You want to convey a high-level summary of the issue to provide a general idea on the scope of the problem.​
  • Craft this carefully to make searching for bugs easier. ​
  • OK to include the top-of-the stack function in case of exception (helps with internal searches)​
What to avoid:​
  • Spelling mistakes? Correct them.​
  • No information that might identify the customer (hostname, IP address, password, username, etc.)​
  • Internal codename (e.g. Lindt)? Remove it.​
  • Do not mention XR release in Headline unless only a single XR release is confirmed to be exposed​
  • See the word “crash”? Replace it with “traceback and reload” or “process exception and restart”​

NONE​
  • Default value​
  • Select this unless you have a very good reason to choose one of the $$ flags​
$$IGNORE​
  • Only used for bugs that customers will never see.
  • Reason for suppressing the Release-note must be clearly explained in Release-note enclosure.​
  • Example:​
    Bugs related to makefile or other changes that affect how images are compiled.​
$$PREFCS​
  • Used for bugs that exist in a feature or functionality that has never been released to customers ​​
  • If a released feature is tested in an unreleased software, do not use the $$PREFCS flag.​​
      • $$IGNORE-PSIRT
      • This flag is added when you explicitly want to hide a bug because you believe it is a Security Vulnerability in our product. After doing so, you must notify psirt@cisco.com of the bug​​
      • Additionally, the Product Security Incident Response Team (PSIRT) will add this flag to all bugs they are reviewing for potential security vulnerabilities. If the bug is deemed not a vulnerability, then PSIRT will remove the flag and release the bug. If it is deemed a vulnerability, then PSIRT will remove the flag ONLY when they publish an external Advisory. ​​
      • DO NOT remove this flag once it has been set. Only PSIRT will remove this flag.​​

What to consider:​
  • "Conditions" section should describe the Conditions, Trigger and Impact.​
Conditions:​
  • Describes the customer environment under which the problem has occurred or might occur.​
  • Examples: specific Hardware or Software configurations, network conditions, type of packet flows​
  • Clearly state whether any or all listed conditions must be met to hit this bug​
Trigger:​
  • Describes the events that may trigger the problem​​
  • Examples: polling of a specific SNMP OID; Primary path flap; etc.​​
Impact:​
  • Describes the impact on the device operations, protocol behaviour, network services.​
  • Quantify the impact when possible​​​
  • Examples:​
    one execution of command X leaks 32kB of heap memory;​
    peer flaps every 10s;

What to consider:​
  • ”Workaround" section should describe the Workaround and Recovery steps. ​
  • Workaround is a proactive action that can be taken to avoid hitting the bug​​
  • Recovery steps restore functionality once the bug is hit​
What to avoid:​
  • Spelling mistakes? Correct them.​
  • No information that might identify the customer (hostname, IP address, password, username, etc.)​
  • “router reload” is not a workaround. Customers get upset when this is the only content of the Workaround section. Use this instead:​
    Workaround:​
    None. Reload router to recover.
